작성 일시: `2026-04-10 21:45:10 KST`

이 문서는 Coreline Orchestrator 저장소를 오케스트레이터 관점에서 실제 구현 가능한 순서로 정리한 작업 문서다. 기존 설계 문서를 구현 로드맵으로 재구성하고, 파일별 착수 순서와 각 Phase의 자체 테스트 기준을 고정한다.

## 개발 목적

Coreline Orchestrator를 문서 중심 스캐폴딩 상태에서 벗어나, 다음을 수행하는 v1 control plane으로 단계적으로 구현한다.

- Job 생성/조회/취소/재시도
- CodexCode CLI worker process spawn 및 lifecycle 관리
- write-capable task의 worktree 격리
- 로그/이벤트/결과 수집 및 집계
- Hono 기반 HTTP API + SSE 제공
- `.orchestrator/` 기반 파일 영속화
- 재시작 이후 reconciliation 및 cleanup

## 문서 기반 제약 사항

- TypeScript ESM + `.js` import extension 유지
- `any` 타입 사용 금지
- 테스트는 `bun:test` 기준으로 작성
- API prefix는 `/api/v1`
- 에러 응답은 `{error: {code, message, details}}`
- write-capable task 기본 격리는 `git worktree`
- worker 실행 명령의 핵심 옵션은 `--print --bare --dangerously-skip-permissions --output-format stream-json --no-session-persistence`
- worker 결과는 orchestrator가 제공한 `ORCH_RESULT_PATH`에 JSON으로 기록
- terminal state는 worker exit callback에서 최종 확정
- atomic file write는 temp file → fsync → rename 순서를 따른다
- v1 집계 정책은 strict failure aggregation을 유지한다

## 공통 진행 규칙

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.

- 구현 중 발생한 이슈는 해당 Phase에 기록하고 그 Phase 안에서 해결한다.

## Critical Path 요약

1. Core contract가 먼저 완성되어야 storage / runtime / API 타입 경계가 고정된다.
2. Storage + Isolation이 있어야 worker lifecycle을 안전하게 시작할 수 있다.
3. Runtime + Logs가 있어야 실제 worker spawn과 진단이 가능하다.
4. Worker lifecycle이 안정화되어야 scheduler와 API가 의미를 가진다.
5. API 이후에 reconciliation/shutdown을 붙여야 운영 가능한 control plane이 된다.

## 잔여 리스크 / 후속 과제

- `codexcode` 실제 실행 표면이 바뀌면 invocationBuilder와 runtime adapter가 가장 먼저 영향을 받는다.
- worktree 테스트는 실제 git fixture 관리가 필요하므로 테스트 유틸 설계가 중요하다.
- API와 scheduler를 너무 일찍 붙이면 진단이 어려워지므로 worker lifecycle 안정화 전까지는 내부 서비스 레벨 테스트를 우선한다.
- v2 기능(session-aware, WebSocket, SQLite)은 v1 안정화 이후 별도 workstream으로 분리한다.
