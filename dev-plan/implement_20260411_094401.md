이 문서는 Coreline Orchestrator v1 완료 이후 발견된 P0 리스크 3개를 최소 범위 hardening patch로 수습하기 위한 작업 문서다. 새 기능 확장이 아니라, v1 동작의 상태 정합성 / trust boundary / 프로세스 제어 공백을 우선 보정하는 데 목적이 있다.

## 개발 목적

다음 3개 P0 리스크를 운영 전 hardening patch로 수정한다.

- terminal job이 `cancel` 요청으로 `canceled`로 덮어써질 수 있는 상태 정합성 버그 제거
- orchestrator 재시작 후 runtime handle이 없는 live PID를 stop하지 못하는 제어 공백 보완
- artifact API가 worker-authored path를 과신해 repo 밖 파일을 노출할 수 있는 trust boundary 취약점 차단

## 개발 범위

- `WorkerManager.stopWorker()` 또는 동등한 runtime stop 경로에 PID fallback terminate 규칙 추가

## Phase 1. Cancel State Hardening

### 목표

terminal job은 더 이상 cancel 요청으로 상태가 바뀌지 않도록 막고, API는 규격화된 409를 반환한다.

### 구현 태스크

- [x] `Scheduler.cancelJob()`에서 `completed/failed/canceled/timed_out` job의 처리 규칙을 명확히 분기한다.
- [x] 이미 terminal인 job에 대해 `InvalidStateTransitionError` 또는 동등한 409 경로를 반환한다.
- [x] cancel success 경로는 non-terminal job에서만 유지한다.
- [x] API 테스트에 completed/failed job cancel → 409 케이스를 추가한다.
- [x] shutdown 경로가 terminal job을 재취소하지 않도록 영향 범위를 확인한다.

### 자체 테스트

- [x] `POST /api/v1/jobs/:jobId/cancel` on completed job → 409 + `INVALID_STATE_TRANSITION`
- [x] `POST /api/v1/jobs/:jobId/cancel` on failed job → 409 + `INVALID_STATE_TRANSITION`
- [x] running/queued job cancel success 회귀 없음
- [x] shutdown test 회귀 없음

### 이슈 및 수정

- [x] terminal job cancel이 silent overwrite 되던 경로를 state-machine error로 승격해 API 409와 정합성을 맞춤

### 완료 조건

- [x] 구현 완료
- [x] 자체 테스트 완료
- [x] 다음 Phase 진행 가능

## Phase 2. Handle-less Live PID Stop Hardening

### 목표

재시작 이후 runtime handle이 없어도, PID가 살아 있는 worker는 최소한 terminate 시도할 수 있게 만든다.

### 구현 태스크

- [x] 현재 `WorkerManager.stopWorker()`의 handle 없는 경로를 재검토한다.
- [x] `worker.pid`가 있고 프로세스가 살아 있으면 `SIGTERM -> timeout -> SIGKILL` fallback 경로를 추가한다.
- [x] handle 없는 PID stop 결과를 worker/job terminalization과 일관되게 연결한다.
- [x] reconcile 이후 live PID를 가진 worker stop 시나리오 테스트를 추가한다.
- [x] restart 이후 stale metadata와 live PID 간 충돌 케이스를 문서화한다.

### 자체 테스트

- [x] runtime handle 없이 `pid`만 있는 active worker를 stop 요청하면 실제 프로세스가 종료된다.
- [x] handle 없는 dead PID는 orphan/lost 정리만 하고 예외 없이 종료된다.
- [x] 기존 handle 기반 stop 경로 회귀 없음
- [x] shutdown 중 handle-less live PID도 정리된다.

### 이슈 및 수정

- [x] runtime handle이 없는 live PID는 상태만 `canceled`로 바꾸지 않고, PID 생존 여부를 확인한 뒤 best-effort terminate 후 동일한 finalize 경로로 수렴시킴
- [x] dead PID 또는 이미 종료된 프로세스는 예외를 던지지 않고 fallback finalize만 수행하도록 분기함
- [x] handle 없는 stop 경로에서도 worktree cleanup이 빠지지 않도록 finalize 단계에서 best-effort cleanup을 보강함

### 완료 조건

- [x] 구현 완료
- [x] 자체 테스트 완료
- [x] 다음 Phase 진행 가능

## Phase 3. Artifact Path Sandbox Hardening

### 목표

artifact API는 repo root 또는 orchestrator-managed 경로만 제공하도록 제한한다.

### 구현 태스크

- [x] artifact path resolve 전에 absolute path / traversal / repo 밖 경로를 검증한다.
- [x] 허용 정책을 명확히 한다: `repoPath` 하위 상대 경로 또는 orchestrator synthetic artifact만 허용.
- [x] `resolveArtifact()`에서 unsafe path는 not found 또는 forbidden 정책으로 차단한다.
- [x] artifact metadata/content API 테스트에 repo 밖 절대 경로, `../` traversal 케이스를 추가한다.
- [x] synthetic artifact(`job_result:*`, `worker_result:*`, `worker_log:*`)는 계속 동작하도록 보장한다.

### 자체 테스트

- [x] repo 내부 상대 artifact path는 정상 조회된다.
- [x] absolute path artifact는 차단된다.
- [x] `../` traversal artifact는 차단된다.
- [x] synthetic artifact 조회 회귀 없음

### 이슈 및 수정

- [x] worker-authored artifact path는 relative-only 정책으로 제한하고 absolute path는 즉시 `ARTIFACT_ACCESS_DENIED`(403)로 차단함
- [x] `..` segment가 포함된 traversal 시도는 파일 존재 여부와 무관하게 resolve 이전에 차단함
- [x] macOS `/var` ↔ `/private/var` canonical path 차이로 safe artifact가 오탐 차단되던 문제를 `realpath(repoRoot)` 비교로 보정함
- [x] synthetic artifact는 계속 허용하되 API에 노출되는 `path`는 repo-relative orchestrator 경로로 정규화함
