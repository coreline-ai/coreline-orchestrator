## Phase 1. Runtime Recovery & Restart Semantics

### 목표

orchestrator 재시작 이후에도 live execution을 어떻게 분류/종료/복구할지 일관된 규칙을 확정한다.

### 권장 정책 결론 (착수 전 잠금)

- [x] **process-mode v1.1에서는 runtime reattach를 지원하지 않는다.**
- [x] live PID + runtime handle 없음 → `terminate_only` 분기로 취급한다.
- [x] dead PID 또는 PID 없음 + non-terminal worker → `lost` 또는 `canceled(created)`로 recovery finalize 한다.
- [x] `POST /workers/:id/restart`는 **실행 unit 재시작이 아니라 terminal worker를 기준으로 새 retry job을 만드는 API**로 명시한다.
- [x] 기존 endpoint path는 유지하되, 응답/문서에서 restart semantics를 `retry_job_clone` 성격으로 드러낸다.

### 구현 태스크

- [x] process-mode worker의 persisted runtime identity 범위를 정의한다 (`pid`, `startedAt`, optional runtime token 등).
- [x] startup reconcile 시 `reattach 가능 / terminate only / lost 처리` 분기표를 문서와 코드 기준으로 맞춘다.
- [x] `POST /workers/:id/restart`가 실제 restart인지 job-level retry인지 API 의미를 고정한다.
- [x] handle-less live PID stop 이후 stale metadata와 recovered state 충돌 규칙을 정리한다.
- [x] 향후 full session-aware runtime으로 이어질 seam은 남기되, 이번 범위에서는 process-mode follow-up만 다룬다.

## Phase 2. File Store Read-Path Hardening

### 목표

### 구현 태스크

- [x] `jobs`, `workers`, `events`, `artifacts` 조회의 병목 경로를 계측/정리한다.
- [x] 최소 범위 index 또는 summary cache 전략을 정의한다.
- [x] `metrics`와 list API가 전체 디렉토리 scan 없이 동작하도록 보강 범위를 정한다.
- [x] index rebuild / partial corruption / cleanup 이후 재생성 규칙을 정리한다.
- [x] SQLite 전환은 제외하되, 향후 migration seam을 해치지 않는 구조로 제한한다.

### 구현 내용

- [x] `FileStateStore`에 `jobs`, `workers`, `artifacts` persistent index(`.orchestrator/indexes/*.json`)를 추가했다.
- [x] index는 startup `initialize()` 시 authoritative JSON/result 파일 기준으로 재생성되며, runtime read path는 in-memory cache를 사용한다.
- [x] `getJob`, `listJobs`, `getWorker`, `listWorkers`가 directory full-scan 대신 cache/index 경로를 사용하도록 변경했다.
- [x] explicit artifact lookup을 위해 `StateStore.findArtifactReference()`와 artifact index를 추가했다.
- [x] `src/api/routes/artifacts.ts`는 synthetic artifact는 direct `getJob`/`getWorker`, explicit artifact는 indexed lookup을 사용하도록 보강했다.
- [x] `listEvents()`는 NDJSON 전체 재파싱을 피하기 위해 file stat 기반 parsed event cache를 사용한다.
- [x] artifact index miss 시 authoritative result 파일 기준 rebuild fallback을 수행해 stale external writes에도 복구 가능하게 했다.

## 최종 결과 요약

- 본 문서는 P0 이후 남은 P1/P2 hardening 항목을 **5개 ordered workstream**으로 고정한다.
- 현재 시점에서는 **Phase 1~5 전체 완료**이며, 다음 우선순위는 v2 planning 또는 manual real-runtime smoke 확장이다.
- 새 기능 확장이 아니라 **운영 안정성, 재현성, 노출 통제, 실환경 검증** 보강이 목적이다.

## 잔여 리스크 / 후속 과제

- full session-aware runtime은 여전히 v2 범위로 남는다.
- SQLite 전환은 read-path hardening 이후 별도 결정이 필요하다.
- auth/RBAC를 외부 multi-tenant 수준까지 확장하는 것은 본 backlog 밖이다.
- distributed leader election / multi-instance coordination도 본 backlog 밖이다.
