# implement_20260411_120538.md

작성 일시: `2026-04-11 12:05:38 KST`

이 문서는 Coreline Orchestrator의 v1 구현, post-v1 hardening, 운영 smoke 검증 이후 다음 주력 workstream인 v2 범위를 고정하기 위한 작업 문서다. 이번 문서는 session-aware runtime, SQLite state store, WebSocket transport를 중심으로 구현 순서를 잠그고, v1 안정성을 해치지 않는 확장 경계를 정의한다.

## 개발 목적

다음 v2 목표를 구현 가능한 순서로 고정한다.

- process-mode 중심 v1에서 session-aware runtime으로 확장할 수 있는 실행 모델 추가
- file-backed state store에서 SQLite-backed store로 확장 가능한 저장 계층 도입
- SSE 중심 v1 transport에 더해 WebSocket 기반 live streaming / attach control 추가
- 기존 v1 API / artifact sandbox / auth hardening을 유지하면서 migration path를 제공

## 개발 범위

- session record / attach-detach / cancel semantics 설계 및 구현
- runtime adapter를 process-mode + session-aware mode로 공존시키는 추상화 보강
- SQLite schema, DAO, migration/import path, bootstrap cutover 구현
- WebSocket event/control transport 추가
- 운영 smoke / migration / rollback / release 절차 문서화

## 제외 범위

- distributed multi-host orchestration
- leader election / shared queue / HA scheduler
- multi-tenant RBAC / SSO / per-user auth
- browser UI / admin console 구현
- object storage artifact registry 전면 도입
- codexcode 자체 수정

## 공통 진행 규칙

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- v1 process-mode 계약은 additive 방식으로만 확장한다.
- storage, runtime, transport를 동시에 크게 바꾸지 않고 seam을 먼저 잠근다.
- migration이 필요한 변경은 import / rollback 경로를 같은 Phase에서 정의한다.
- 문서와 테스트 없는 신규 API는 추가하지 않는다.
- hardening에서 잠근 auth / artifact / recovery 정책은 후퇴시키지 않는다.

## 문서 기반 제약 사항

- `resultPath` 기반 worker-authored structured result contract는 유지한다.
- artifact API는 repo-relative / orchestrator-managed sandbox 규칙을 유지한다.
- `untrusted_network` 모드의 token auth와 redaction 정책은 유지한다.
- `/api/v1/*`는 호환성을 유지하고, v2 확장은 additive field 또는 신규 endpoint로만 도입한다.
- session-aware runtime도 `RuntimeAdapter` seam 뒤에 위치해야 하며 scheduler가 runtime별 분기를 직접 품지 않도록 유지한다.

## 권장 구현 순서 요약

1. **Contract 먼저** — session/API/store/transport 경계를 잠근다.
2. **Runtime 다음** — session-aware adapter와 recovery semantics를 추가한다.
3. **Store 전환** — SQLite를 도입하되 import/fallback을 같이 만든다.
4. **Transport 확장** — WebSocket을 SSE 위에 additive하게 추가한다.
5. **운영 검증** — manual smoke, migration, rollback까지 묶어 ship 판단을 한다.

## 최종 결과 요약

- v2는 단일 큰 리라이트가 아니라 `runtime -> store -> transport` 순의 staged upgrade로 진행한다.
- v1 process-mode, artifact sandbox, external auth/redaction은 v2에서도 유지되어야 한다.
- session-aware runtime, SQLite, WebSocket은 모두 migration/rollback 경로와 함께 들어가야 한다.

## 잔여 리스크 / 후속 과제

- session-aware runtime의 실제 codexcode capability와 attach semantics가 CLI 릴리스에 따라 달라질 수 있다.
- SQLite cutover 시 file-store import 성능과 locking 전략을 별도로 검증해야 한다.
- WebSocket 도입 후에도 multi-host / RBAC / audit trail은 별도 workstream으로 남는다.
