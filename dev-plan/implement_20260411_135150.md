이 문서는 v2 Phase 1~5 완료 이후 Coreline Orchestrator의 다음 개발 우선순위를 고정하기 위한 post-v2 작업 문서다. 이번 문서는 현재 코드베이스의 아키텍처 상태를 기준으로, session runtime을 진짜 interactive continuation 단계까지 끌어올리고 이후 auth/audit/distributed execution으로 넘어가기 위한 순서를 잠근다.

## 개발 목적

다음 post-v2 목표를 구현 가능한 순서로 고정한다.

- lifecycle/session record 수준에서 멈춘 현재 v2를 **true interactive session runtime**까지 확장한다.
- process/file/sqlite/websocket 위에 올라간 단일 호스트 control plane을 **운영-grade auth/audit** 단계로 끌어올린다.
- single-process/single-host 가정을 유지한 현재 구조를 **distributed-ready seam**까지 분해한다.
- 이후 multi-host executor 또는 remote worker plane으로 갈 수 있는 migration 순서를 명시한다.

## 개발 범위

- `RuntimeAdapter` 뒤에 reattach 가능한 session-aware adapter 추가
- session input/output streaming, transcript persistence, reconnect semantics 구현
- operator/admin diagnostics, audit trail, session transcript inspection 보강
- shared token 기반 노출 제어를 service token / scoped auth / audit 단계로 강화
- scheduler/state/runtime seam을 distributed-ready 구조로 분리하는 계획 및 구현

## 현재 아키텍처 진단 요약

### 강점

- `core / storage / runtime / workers / scheduler / api / reconcile` 계층 분리가 유지되고 있다.
- file + sqlite dual backend, SSE + WS dual transport, process + session dual execution mode가 모두 additive하게 올라가 있다.
- hardening 이후 artifact sandbox, token auth/redaction, recovery semantics가 비교적 안정적이다.
- v2 Phase 5까지 smoke / migration / release gate가 문서와 코드로 고정되었다.

### 핵심 한계

- 현재 session은 **record/lifecycle/control** 수준이지 **interactive continuation runtime**이 아니다.
- `SessionManager`와 WebSocket route는 붙었지만, 실제 session I/O를 중계하는 `SessionWorkerClientAdapter`는 아직 없다.
- auth는 여전히 shared token 모델이며, operator identity / audit trail / repo scope authorization이 없다.
- scheduler/event bus/queue가 in-memory 중심이라 distributed execution으로 바로 확장되지는 않는다.
- state backend는 file/sqlite 단일 호스트 전제이며 multi-writer 동시성 모델이 없다.

### 다음 우선순위 결론

가장 먼저 해야 할 일은 **Phase 1 — True Session Runtime & Reattach** 다.

이유:
- 현재 문서상 가장 큰 architectural gap이 “session mode promises reattach, but runtime adapter is still not truly session-aware” 이기 때문이다.
- 이 갭을 메우지 않으면 WebSocket/session API가 lifecycle façade에 머물고, 이후 auth/distributed work도 실제 사용자 가치와 멀어진다.

## Phase 1. True Session Runtime & Reattach

### 목표

현재 session lifecycle façade를 실제 same-session continuation 가능한 runtime으로 전환한다.

### 설계 잠금 결론

- `session` mode만 **same-session reattach** 를 보장한다.
- `background` mode는 long-lived process일 수 있어도 **interactive continuation** 을 약속하지 않는다.
- `process` mode는 기존 v1/v2 계약을 유지하며, 이번 Phase의 변경은 additive하게만 확장한다.
- `RuntimeAdapter` seam 밖으로 session transport 세부 구현이 새지 않도록 유지한다.
- transcript persistence와 transcript 조회 API는 이번 Phase의 핵심 완료 조건이 아니며, 최소한의 runtime cursor / backpressure metadata까지만 다룬다.
- session interactive transport의 1차 표면은 `WS /api/v1/sessions/:sessionId/ws` 로 유지하고, SSE는 passive observation 용도로 유지한다.

### 구현 태스크

- [x] `RuntimeAdapter`에 session attach/detach/sendInput/readOutput capability를 추가한다.

- [x] `SessionRecord`에 runtime identity/session transcript cursor/backpressure metadata를 확장한다.
- [x] orchestrator restart 후 same-session reattach path를 구현한다.
- [x] `/api/v1/sessions/:id/ws`를 lifecycle control에서 interactive runtime transport까지 확장한다.

## 권장 구현 순서 요약

1. **Session runtime gap 해소** — 현재 가장 큰 architectural mismatch를 먼저 닫는다.
2. **Interactive ops 고정** — transcript/replay/diagnostics 없이 long-lived session을 운영하지 않는다.
3. **Auth/Audit 강화** — operator identity와 권한 경계를 명확히 한다.
4. **Distributed seam 분리** — queue/store/runtime abstraction을 먼저 분리한다.
5. **Multi-host 검증** — 마지막에만 remote executor MVP로 간다.

## 최종 결과 요약

- post-v2의 핵심 과제는 “더 많은 endpoint”가 아니라 **session runtime의 실질화**다.
- 그 다음은 **auth/audit 강화**와 **distributed-ready seam 분리**다.
- multi-host execution은 지금 바로 구현할 대상이 아니라, Phase 1~4를 통과한 뒤 제한된 MVP로 검증해야 한다.

## 잔여 리스크 / 후속 과제

- shared token 모델을 오래 유지하면 operator auditability 부족이 누적된다.
- same-session reattach를 구현하지 않은 채 session surface를 더 넓히면 API 의미가 다시 흐려질 수 있다.
- distributed work를 너무 일찍 시작하면 현재의 안정적인 single-host path를 깨뜨릴 수 있다.
