작성 일시: `2026-04-11 21:07:12 KST`

이 문서는 post-v2 Phase 1~5 완료 이후 Coreline Orchestrator의 다음 개발 범위를 고정하기 위한 작업 문서다. 이번 문서는 초기 multi-host prototype을 넘어, shared SQLite coordinator/queue, polling event replay, manifest transport, remote worker-plane fencing, distributed failover verification을 갖춘 다음 단계의 distributed control plane 개발 순서를 잠근다.

## 개발 목적

현재 multi-host prototype을 운영 가능한 distributed control plane 방향으로 확장하기 위한 다음 우선순위를 구현 가능한 순서로 고정한다.

- in-memory coordinator / local queue / local event bus 의존을 external/shared seam으로 치환할 수 있게 한다.
- shared filesystem 전제를 제거하고 artifact/log/result/transcript transport를 remote-friendly contract로 전환한다.
- lease-based single-leader prototype을 host-loss와 reconnect를 견디는 control plane으로 끌어올린다.
- remote executor가 동일 계약으로 claim/heartbeat/result publish를 수행할 수 있게 worker-plane contract를 실질화한다.
- 최종적으로 single-host v1/v2 계약을 유지한 채 distributed 모드를 additive하게 ship할 수 있도록 cutover 절차를 만든다.

## 개발 범위

- external/shared coordinator contract 및 backend seam 확장
- shared queue / shared event stream / replay contract 도입
- object-store manifest 기반 artifact/log/result/transcript transport 도입
- remote executor claim/heartbeat/result publish 경로 구체화
- multi-host failover / fencing / replay / rollback smoke 및 ops 문서화

## 공통 진행 규칙

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- distributed 관련 변경도 v1/v2 single-host 계약을 깨지 않는 additive 방식으로만 반영한다.
- `resultPath` 기반 worker-authored result JSON contract는 유지한다.

- 문서, smoke, rollback 절차 없이 새 operator surface를 추가하지 않는다.

## 문서 기반 제약 사항

- `session` mode의 same-session reattach 보장은 유지하되, distributed 전환 후에도 process/background 계약을 암묵적으로 확장하지 않는다.
- external/shared coordinator는 durable domain state를 직접 대체하지 않고, lease/heartbeat/ownership hint 중심으로 시작한다.
- queue/event bus externalization 전까지도 `StateStore`는 jobs/workers/sessions/results/events/transcripts의 authoritative source를 유지한다.
- remote artifact/log/result transport는 repo-relative/orchestrator-managed sandbox 정책을 보존해야 한다.
- `untrusted_network` 노출은 named/shared token auth, scope auth, audit trail을 유지해야 한다.

## 현재 아키텍처 진단 요약

### 강점

- session/sqlite/ws/auth/audit/distributed-ready seam까지 이미 additive하게 올라가 있다.
- lease-based single-leader multi-host prototype으로 scheduler failover 방향이 검증됐다.
- `src/control/remotePlane.ts`에 remote worker-plane 최소 계약이 고정돼 있다.
- smoke/migration/release gate 문화가 이미 잡혀 있어 다음 단계도 operator-oriented하게 검증할 수 있다.

### 핵심 한계

- coordinator backend는 shared SQLite prototype까지 올라왔지만, production-grade external coordinator service는 아직 없다.
- event stream은 `state_store_polling` 기반이라 broker-pushed fan-out/backpressure 제어는 아직 없다.
- `object_store_manifest` transport는 shared filesystem blob backing store를 전제로 하며 true network object store cutover는 아직 없다.
- remote executor contract는 fencing metadata까지 포함하지만 실제 network worker-plane protocol은 아직 local simulation 수준이다.
- cluster-wide failover smoke는 통과했지만 distributed cutover/rollback 운영은 아직 prototype boundary를 벗어나지 않았다.

### 다음 우선순위 결론

이번 문서 범위의 **Phase 1 ~ Phase 5 구현과 검증은 모두 완료**되었다.

완료 기준:
- shared SQLite coordinator / queue, polling event stream, manifest transport, remote worker-plane fencing metadata, distributed smoke / release gate가 모두 반영되었다.
- `bun test`, `bun run build`, `bun run ops:smoke:multihost:prototype`, `bun run ops:verify:distributed`까지 통과했고 `release:distributed:check` composed gate가 추가되었다.
- 다음 우선순위는 production-grade external coordinator service / broker-backed queue-event stream / network object-store cutover를 다루는 새 로드맵이다.

## Phase 1 세부 설계 잠금

### 잠금 결론

- external/shared coordinator는 **durable domain state 저장소가 아니라 ephemeral ownership/lease/heartbeat 계층**으로 유지한다.
- `StateStore`는 jobs/workers/sessions/events/results/transcripts의 authoritative source를 계속 유지한다.
- fencing token은 최소한 **dispatch lease** 와 **worker assignment** 두 축에 대해 stable/monotonic 하게 표현되어야 한다.
- single-host fallback은 유지하며, `InMemoryControlPlaneCoordinator`는 앞으로도 reference implementation 역할을 계속한다.
- external coordinator backend 선택(예: Redis/DB/service)은 이번 Phase의 범위가 아니며, 먼저 contract와 takeover 규칙만 잠근다.
