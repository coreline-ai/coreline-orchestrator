# implement_20260412_160606.md

작성 일시: `2026-04-12 16:06:06 KST`

이 문서는 Coreline Orchestrator가 현재까지 ship한 distributed/service control-plane 기능을 실제 운영 가능한 production baseline으로 끌어올리기 위한 다음 개발 순서를 고정하고, 이후 작업이 다시 prototype 확장이나 문서 드리프트로 흐트러지지 않도록 하기 위한 작업 문서다.

## 개발 목적

현재 저장소는 단일 호스트 v1, v2 session/SQLite/WebSocket, post-v2 distributed seams, service-backed control plane path, remote executor network worker-plane smoke, manual/deep verification follow-up까지 완료했다.

다음 단계의 목적은 다음과 같다.

- production 운영에 필요한 **실제 backend/provider contract**를 고정한다.
- **관찰성 / SLO / alerting / audit depth**를 운영 기준으로 강화한다.
- **executor identity / secret rotation / transport auth**를 production-grade로 보강한다.
- **load / soak / chaos / canary / rollback automation**을 다음 ship gate로 정착시킨다.
- 향후 정식 release를 위한 **GA(General Availability) readiness 기준**을 문서와 스크립트로 고정한다.

## 개발 범위

- external coordinator / queue / event / object store provider contract refinement
- production observability, SLI/SLO, alerting, audit enrichment
- executor/service identity, credential rotation, transport auth hardening
- load / soak / fault-injection / canary / rollback automation
- GA readiness checklist, release gate, operator automation

## 공통 진행 규칙

- 현재 `file/sqlite/service` fallback과 single-host contract는 명시적 cutover 전까지 유지한다.
- distributed backend/provider 작업은 항상 rollback path, degraded-mode path, smoke path를 함께 제공한다.
- 모든 운영성 강화는 script/test/문서 형태로 남겨 재현 가능해야 한다.
- auth/audit/redaction 규칙은 backend/provider가 바뀌어도 후퇴시키지 않는다.
- 각 Phase는 자체 테스트와 문서 업데이트까지 포함해 완료 처리한다.
- 체크박스 상태를 실제 진행 상태와 맞게 유지한다.

## 잔여 리스크 / 후속 과제

- 현재 service-backed distributed path는 ship됐지만, 실제 production provider 운영에서는 provider별 latency/failure semantics 차이가 커질 수 있다.
- Bun exit-delay는 workaround/probe가 준비됐지만, 런타임 업그레이드마다 재검증이 필요하다.
- deep verification cadence는 정해졌지만, 실제 scale/load baseline은 아직 조직/환경 기준으로 더 수치화가 필요하다.
